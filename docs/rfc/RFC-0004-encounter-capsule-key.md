# RFC-0004: 遭遇カプセル鍵クラスの固定（OPEN-007）

## Status
Accepted
Owner: agent1
Date: 2026-02-05

## Context

L1（Share Envelope）実装前の未決 OPEN のうち、OPEN-007（遭遇カプセルの暗号化鍵を誰が持つか）が暗号境界を確定できずブロッカーになっていた。

候補は以下の 3 つだった。

- 救助機関鍵（Authorized Rescue key）
- 地域鍵
- 家族鍵

この判断は「鍵のクラス」を固定するためのものとし、Authorized Rescue の運用主体や組織定義（OPEN-010）は別問題として分離する。

## SR（推奨案）

**遭遇カプセルは Authorized Rescue 公開鍵で暗号化する。**

## 理由

1. **救助可用性を主ルートで担保できる**
   - 家族不在・連絡不能でも救助側が復号可能で、孤立者救助の初動を止めない。
2. **運用の現実性が高い**
   - 地域鍵で必要になる配布/ローテーション/失効/漏洩時対応を、L1 の前提に持ち込まずに済む。
3. **relay/reveal 境界と整合する**
   - 通行人端末は暗号化カプセルを運ぶだけ（relay-only）で復号できない。
4. **OPEN-010 と責務分離できる**
   - 先に鍵クラスだけ固定し、運用主体の定義は OPEN-010 で継続審議できる。

## 却下案と理由

- **地域鍵**
  - 鍵配布境界が地域運用に依存し、広域災害時にローテ/失効が破綻しやすい。
  - 漏洩時の影響半径が広く、緊急時オペレーションが重い。
- **家族鍵**
  - 家族不在・関係断絶・連絡不能時に復号経路が詰まる。
  - 救助機関の一次対応より先に家族鍵依存が発生し、主目的（救助初動）を阻害する。

## OPEN-010 との境界

- この RFC で確定するのは **鍵クラス**（Authorized Rescue key）まで。
- 以下は **未決のまま維持**し、OPEN-010 で扱う。
  - Authorized Rescue の定義（組織・権限主体）
  - 鍵配布/ローテーション/失効/監査運用
