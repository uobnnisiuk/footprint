# DEC-0006: Transport Scope（Multi-Transport Attempt）不採用

## 日付
2026-02-05

## 決定内容（SR）

**Transport Scope（複数通信手段を"全部試す"を主軸に据える方針）は、不採用とする。**
- Transport Scopeを中核方針（Guaranteeの根拠）として採用しない。
- 検討過程で作成した Transport Scope 関連資料（例: RFC-0006 / REV-0006）は、**未コミットのまま撤回**し、SSOTには残さない（本DECのみ記録として残す）。

## 理由

1. **保証の後退**: 「外部に手がかりが出る」保証が「試行するだけ」へ後退し、footprintの約束（救助の手がかりを残す）と緊張関係が生じる。
2. **当初主軸（BLE P2P）からの逸脱**: iOS制約下でBGすれ違いP2Pの成立保証が置けず、P2Pを主軸にした設計思想が保証ラインから外れる。
3. **検証不能化リスク**: 多経路の状態空間が爆発し、端末/OS/前景後景/電波/電池の掛け算で再現性が低下する。正確性重視・CI green維持の開発スタイルと相性が悪い。
4. **悪用耐性未確定**: relay(B)（遭遇カプセル中継）をスコープに含めるにはスパム/偽カプセル対策（OPEN-008）が核心だが、未解決のままでは前進できない。
5. **オフライン救助ストーリー未確定**: プラットフォーム不在時のT2補完（OPEN-012）が未確定のままでは、災害時にインフラが死んだ場合の主経路が閉じない。

## 影響

- Transport Scope を前提とした「Tier設計」「多経路オーケストレーション」等の仕様は採用しない。
- 既存のRISK（iOS BLE制約など）自体は残し、別の形で保証ラインを再設計する（本DECでは代替案は確定しない）。

## 再検討条件（必要十分のみ）

- iOS/Android双方について「操作不能でも外部に観測されうる最小の手がかり（Minimum External Evidence）」の保証ラインが、Yes/Noで判定できる形で仕様化されること。
- OPEN-008（スパム対策）および OPEN-012（T2補完）の解決方針が確定し、検証計画に落ちること。

## 関連

- 関連RISK: iOS BLE制約により「操作ゼロの痕跡/中継」をスマホ単体で保証できない
- 関連OPEN: OPEN-008（スパム対策）, OPEN-012（プラットフォーム不在時のT2補完）
